## 🗳️ 3. مصادقة حفظ التقدم (Dynamic Progress Persistence Validation)

*   **آلية الحفظ:** تم فحص وتأكيد الشفرة للتحقق من أن تقدم العملية ونقل الملفات لا يُسجّلان في الذاكرة العشوائية للخادم مطلقاً.
*   **مسار تدفق الحالة:**
    1.  يتم إعداد المهمة بوضع `pending` داخل قاعدة بيانات Supabase في جدول `import_jobs`.
    2.  يتولى منسق الشفرة السحابية تفكيك الملف إلى حزم منفصلة (Chunks) بحجم `100` سجل لكل حزمة.
    3.  مع نهاية معالجة كل حزمة برمجية، يحصل جدول `import_chunks` وجدول الداتا الحية على تحديثات فورية برقم الحزمة وحالة الإدراج والتقدم المطرد.
    4.  في حال حدوث كراش أو إيقاف قسري للحاوية (Crash / Restart)، تعاد قراءة الحزم المنجزة سحابياً واستئناف العمل من النقطة الأخيرة الآمنة (No Data Loss).

## 🚀 4. اختبار جهد نظري ومحاكاة إنتاجية ضخمة (Stress Test Projection Model)
### لـ 100,000 منتج (Products) و 1,000,000 حركة مالية وفواتير (Transactions)

فيما يلي محاكاة معمارية دقيقة لأداء المحرك تحت الضغوط القصوى والبيانات الضخمة جداً:

```
[ SQLite File ] ──( 115,000 Streams )──> [ Express ETL Service ] ──( 1,100 Chunks )──> [ Supabase DB / REST API ]
```

### أ) زمن التنفيذ المتوقع والجدولة الزمنية (Expected Execution Duration)
1.  **مرحلة قراءة وتجهيز البيانات (Parsing & Transforming Phase):**
    *   ستأخذ قراءة ملف SQLite ضخم (حوالي 250MB إلى 500MB) وفك ترميز الجداول في Node.js باستخدام `SqliteReader` حوالي **8 إلى 15 ثانية** فقط لأنها عملية تتم كلياً في الذاكرة ومبنية على مستعلم SQLite خفيف.
2.  **مرحلة الترحيل والاتصال السحابي (Ingestion & Network Phase):**
    *   بافتراض حجم حزمة الترحيل الفعلي هو 100 سجل في الحزمة الواحدة:
        *   عدد الحزم للمنتجات: `1,000` حزمة.
        *   عدد الحزم للحركات والفواتير: `10,000` حزمة.
        *   الإجمالي: `11,000` حزمة ترحيل.
    *   بافتراض أن استجابة الشبكة (Network Roundtrip Time) إلى خوادم Supabase تأخذ بالمتوسط 80ms لكل حزمة إدراج دفعة واحدة (Batch Upsert):
        *   **الزمن التقديري الفعلي المباشر:** `11,000 * 0.080s = 880 ثانية` (أي حوالي **14.6 دقيقة**).
    *   عند تفعيل خاصية الاتصال المتعدد المتوازي (Multi-Threaded Parallel Chunks) بعدد **4 ممرات متوازية**، يتقلص الزمن المتوقع إلى **3.6 دقيقة** فقط ترحيل متكامل.

### ب) استهلاك الذاكرة المتوقع (Expected RAM Utilization)
*   **الوضع العادي (Idle Mode):** `35MB to 45MB`.
*   **خلال تشغيل النقل الضخم (Ingestion Spike):**
    *   بما أننا نستخدم "قراءة دفقات جزئية" (Chunk-by-Chunk Stream Ingestion) حيث لا نحمل المليون سجل داخل مصفوفة واحدة في الرام، بل يتم تشغيل المؤخرة وجلب البيانات عبر تقسيمات SQLite المتتالية، فإن استهلاك الذاكرة المتوقع في خادم الـ Node.js سيتأرجح بين **160MB إلى 280MB** كحد أقصى، مما يجعل النظام متوافقاً تماماً مع أرخص مستويات التشغيل بالحاويات (Standard Serverless Containers).

### ج) نقاط الاختناق المحتملة والحلول المعمارية (Potential Bottlenecks & Remedies)

1.  **اختناق الاتصالات بقاعدة البيانات (Database Connection Pool Exhaustion):**
    *   *المسبّب:* توازي عدد كبير من حزم الإدخال الراجعة من الحاويات المختلفة بآن واحد.
    *   *العلاج المعماري:* محرك `BatchImporter` يلتزم بحجم حزم ثابت ويستخدم استدعاءات وحيدة غير متشعبة لتحديد أقصى حيز من قنوات الربط.
2.  **قيود استهلاك واجهات سوبابيس (API Rate Limits):**
    *   *المسبّب:* إرسال الآلاف من طلبات POST/PATCH المتلاحقة خلال فترات ضيقة.
    *   *العلاج المعماري:* اللجوء لأكواد مجمعة داخل قاعدة البيانات (PostgreSQL Store Procedures / RPCs) لتمرير مصفوفة جيسون كبيرة من الصرّاف بطلب بروتوكول شبكة واحد.
3.  **سعة ملفات التخزين المؤقتة المحلية وعزل المساحة المتاحة (Ephemeral Disk Limits):**
    *   *المسبّب:* امتلاء المساحة المتاحة للحاوية عند رفع ملفات ضخمة متعددة.
    *   *العلاج المعماري:* التخلص ومسح ملف SQLite المرفع عن طريق استدعاء ميثود الحذف الذاتي فور نجاح المهمة، مع الإقرار بوجود ملفات هجرة فريدة.
