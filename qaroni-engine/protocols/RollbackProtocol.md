# Qaroni AI Engine — Rollback Protocol Specification
## بروتوكول التراجع والإنقاذ التلقائي وإعادة استقرار العمليات

---

## 1. الغرض والأهداف (Purpose & Scope)
يهدف هذا البروتوكول الفولاذي إلى مأسسة روتينات التراجع والإنقاذ وإلغاء التغييرات الهيكلية والبرمجية في حال رصد أي فشل، تباطؤ، خرق أمني، أو عدم استقرار في بيئات الفحص أو الإنتاج البعدي، بهدف إعادة النظام فوراً لحالة الصفر المستقرة وحماية البيانات.

---

## 2. القيود والحدود الأمنية لعمليات التراجع (Rollback Constraints)
* **التفعيل التلقائي الفوري (Zero-Latency Activation):** عند فشل أي اختبار بعدي أو رصد انهيار في واجهات المستخدم، يتم إطلاق هذا البروتوكول فوراً ودون انتظار موافقة بشرية كإجراء أمني طارئ.
* **حرمة تدمير البيانات (Data-Safe Rollback):** يُمنع استخدام روتينات التراجع التي تتضمن مسحاً حراً للبيانات دون حفظ نسخة احتياطية مسبقة، ويجب أن يقتصر التراجع الهيكلي على الحفاظ التام لسلامة الصفوف الحية.
* **التراجع الشامل المزدوج (DB & Code Alignment):** يجب مزامنة تراجع قاعدة البيانات (Down Script) مع إعادة وتراجع فرع GitHub المعني للحالة والوسم (Tag) السابق لضمان تماسك الكود الهيكلي بالسيرفر.

---

## 3. خطة وخطوات تفعيل البروتوكول (Execution Flow of RollbackProtocol)
1. **الخطوة الأولى: التقاط إشارة الفشل وإيقاف البث (Capture Failure Signal):**
   * إيقاف وتجميد قنوات بوابة الكتابة `WriteGateway` فوراً لمنع تفاقم المشكلة.
2. **الخطوة الثانية: تطبيق تراجع قاعدة البيانات (Apply DB Rollback):**
   * تنفيذ كود التراجع (Down SQL Script) الذي تم توليده ومحاكاته مسبقاً في مرحلة البناء لإرجاع الهياكل والمخططات للحالة السابقة بنسبة 100%.
3. **الخطوة الثالثة: تراجع كود GitHub (GitHub Reset):**
   * إرجاع وإعادة تعيين فرع GitHub للحالة السليمة السابقة وإلغاء الـ Commits المسببة للمشكلة.
4. **الخطوة الرابعة: نشر الحدث والتدقيق الطارئ (Publish Event):**
   * إطلاق الحدث `RollbackCompleted` ونشر تقرير طارئ يحلل أسباب الانهيار وتجميد التذكرة وإرسال إخطار عاجل للمالك.

---
