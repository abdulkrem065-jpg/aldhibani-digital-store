## 2. تفصيل مراحل الأنبوب الـ 16 الحاكم (Detailed Stage Specifications)

### 1. READ (القراءة المبدئية)
يقوم الوكيل `Qaroni_Reader` بالاتصال السطحي وقراءة المخططات والـ Metadata لـ Supabase والتأكد من مطابقة كشف الجداول والملفات المستهدفة بالتعديل.

### 2. VALIDATE (التحقق الدستوري المبدئي)
مطابقة الطلب المطلوب ببنود الدستور الحاكم `EngineConstitution.md` والتأكد من عدم مساسه بالمبادئ الأساسية وحماية البيانات.

### 3. DECISION TRACE (تتبع القرارات الدستورية)
توثيق أثر القرار الدستوري برمجياً وبالمستندات بالصيغة:
`Constitution (Article X) → ADR-XXX → Specification (Module Y) → Action (Migration Z)`.

### 4. RISK ASSESSMENT (تقييم المخاطر وتوجيه البوابات)
تطبيق "بروتوكول تقييم المخاطر" وتصنيف العملية (Low/Medium/High). في حال تصنيف **High** (مثل حذف أعمدة أو جداول، تعديل الـ RBAC)، يتم منع أي تنفيذ تلقائي وتجميد العملية للموافقة اليدوية الإلزامية لاحقاً.

### 5. ANALYZE (التحليل وتحديد الفجوات الفنية)
يقوم الوكيل `Qaroni_Analyzer` بحساب الفروقات الفنية بين الهيكل الحقيقي والـ Specification المطلوب وصياغة كشف دقيق لخطوات الفجوة الهيكلية.

### 6. BUILD (البناء والصياغة المحلية)
يقوم الوكيل `Qaroni_MigrationBuilder` بكتابة ملفات التعديل البرمجية وبناء مسودات الـ Migration (Up/Down) بلغة SQL وتدوين روتينات TypeScript المصاحبة وتخزينها محلياً.

### 7. ISOLATION (تأسيس بيئة العزل)
تفعيل استراتيجية العزل المتدرج بدءاً من الفحص المنطقي للعلاقات (`Knowledge_Validation`) ثم إنشاء فرع مخصص ومستقل لقاعدة بيانات Supabase ليكون مكاناً للفحص المادي.

### 8. TEST (الاختبار والمحاكاة المادية)
تشغيل الفحوصات الفنية على الفرع المعزول: اختبار الصياغة النحوية لـ SQL، سلامة العلاقات والمفاتيح الخارجية (FK)، وجود الفهارس الملائمة (Indexes)، ومحاكاة عمليات الإدخال والاستعلام للتأكد من تفعيل RLS.

### 9. SELF-VALIDATION ENGINE (محرك التحقق والتقييم الذاتي)
مكون برمجي يحلل مخرجات الاختبارات. في حال النجاح المطلق يتم التوجيه للمرحلة 12، وفي حال الفشل يتم نداء حلقة التغذية الراجعة والإصلاح فوراً.

### 10. FEEDBACK & REPAIR LOOP (التغذية الراجعة والإصلاح الذاتي)
عند فشل الاختبار: يقوم المحرك بتحليل الخطأ، وتوليد "تقرير إصلاح" (Fix Report) يوضح التعديل الواجب، وإعادة بناء الكود وإعادة تشغيل الاختبارات. يتاح للنظام **3 محاولات تلقائية فقط** للإصلاح الذاتي.

### 11. ROLLBACK (التراجع والإنقاذ التلقائي)
إذا استنفذت محاولات الإصلاح الـ 3 دون نجاح، أو في حال فشل الفحص البعدي على الإنتاج، يتم إلغاء التعديل فوراً، وتفعيل كود التراجع (Down Script)، وإرجاع الشيفرات البرمجية لنقطة الصفر السليمة وإخطار المالك.

### 12. APPROVAL WAIT (بوابة التفويض البشري)
تجميد الجلسة تلقائياً وعرض كود التحديث ومسودات الـ SQL ونتائج الاختبارات الناجحة ورمز تتبع القرار للمالك، وانتظار الموافقة الصريحة بواسطة رمز الأمان OTP لتفعيل التنفيذ الحقيقي.

### 13. EXECUTE (البث الحي والتنفيذ)
يقوم الوكيل `Qaroni_Executor` بتمرير التحديث وتطبيقه على قاعدة بيانات الإنتاج لـ Supabase كمعاملة واحدة وتنزيل تحديثات الكود بفرع GitHub المعتمد.

### 14. POST-VALIDATION (التحقق والاستقرار البعدي)
إعادة إطلاق اختبارات الاستقرار (Smoke Tests) وفحص جدران الأمان على خادم الإنتاج الحقيقي للتأكد من ثبات واجهات المستخدم وصحة الوصول المعزول.

### 15. CLEANUP (التنظيف الشامل وإغلاق البيئات)
حذف فروع قاعدة البيانات المؤقتة التي تم إنشاؤها للفحص، مسح الرموز والملفات المؤقتة، وتصفير سجلات الذاكرة وإرجاع طاقة المحرك لوضع الاستعداد.

### 16. AUDIT LOG (سجل التدقيق الشامل والنزاهة الفنية)
تدوين تفاصيل المهمة بالكامل شاملة هوية الوكلاء المنفذين، تاريخ المعاملات، تتبع تذكرة القرار الدستوري، ونسبة مطابقة النزاهة وحفظ التقرير في الملف `/Phase1IntegrityAudit.md` ليكون سجلاً أمنياً غير قابل للتحوير.
