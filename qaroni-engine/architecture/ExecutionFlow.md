# Qaroni AI Engine — Execution Flow Specification
## مواصفة تدفق وخطوات تنفيذ العمليات البرمجية والهيكلية

تتبع كافة المهام المسندة لمحرك **Qaroni AI Engine** تدفقاً تنفذياً صامتاً، آمناً، وتدريجياً يتكون من سبع مراحل رئيسية لضمان عدم حدوث أي خطأ برمي أو تلف للبيانات السحابية.

---

## 1. مخطط تدفق العمليات المعياري (Standard Operating Flow Chart)

```
 [START: Created]
        │
        ▼
   [1. Reading] ───────► (Fetch DB Metadata, Read Current Schemas)
        │
        ▼
  [2. Analyzing] ──────► (Evaluate Schema Changes, Identify Dependencies)
        │
        ▼
   [3. Planning] ──────► (Draft Code/Migration Steps in Isolated Sandbox)
        │
        ▼
[4. Validating] ─────► (Simulate Execution, Check RLS & Constrains, Lint)
        │
        ▼
 [WaitingApproval] ───► (Prompt User for Digital Signature / OTP Check)
        │
        ├───[Denied] ──► [TERMINATE: Failed & Logged]
        │
        ▼ [Approved]
   [5. Deploying] ─────► (Apply to Production DB/Git via Write-Gateway)
        │
        ▼
  [6. Verifying] ──────► (Perform Smoke Run, Schema Verification, Post-Audit)
        │
        ├───[Failed] ──► [7. RolledBack] ──► (Trigger Rollback Protocol)
        │
        ▼ [Succeeded]
  [Completed & Logged]
```

---

## 2. تفاصيل مراحل التدفق التشغيلي

### المرحلة 1: القراءة والمعاينة الآمنة (Reading & Ingress Phase)
* يتم تشغيل البروتوكول `ReadProtocol`.
* يقوم الوكيل الحارس `Qaroni_Reader` بالاتصال السطحي السليم بالمخططات الحالية لبيئة Supabase وقراءة الـ Metadata للأكواد ومحتوى مستودع GitHub ذو العلاقة بالمهمة.
* يُمنع منعاً باتاً استخلاص أي بيانات مالية أو ديون أو كلمات مرور الموظفين الحقيقية، والتركيز ينصب فقط على المخططات والهياكل الهندسية.

### المرحلة 2: التحليل وتحديد الفجوات المعمارية (Analyzing Phase)
* يحلل الوكيل المحلل `Qaroni_Analyzer` التغييرات المطلوبة ويقارنها مع الهيكل الحقيقي الحاضر بالاعتماد على بروتوكول `AnalyzeProtocol`.
* يتم بناء شجرة تبعيات كاملة للتعديل المطلوب لتحديد الجداول أو الملفات التي ستتأثر مباشرة بالتغيير، والتنبؤ بأي تعارضات قبل الحدوث.

### المرحلة 3: التوليد والمسودة الافتراضية (Planning & Sandbox Generation)
* يقوم الوكيل المعماري وصانع الترحيلات `Qaroni_MigrationBuilder` بإنشاء الشيفرات البرمجية أو ملفات الترحيل الهيكلية المطلوبة.
* يتم تدوين وكتابة المسودة كلياً داخل الـ Sandbox والذاكرة المحلية للمحرك ويحظر ترحيلها حية.

### المرحلة 4: التحقق والمحاكاة الصارمة (Simulation & Validation Phase)
* يقوم الوكيل الفاحص والمدقق `Qaroni_Validator` بإخضاع المسودة المولدة لسلسلة اختبارات صارمة تشمل:
  * مطابقة الأكواد لأحدث قواعد الـ Linter و TypeScript.
  * فحص جدران الأمان وحوكمة سياسات RLS ومطابقة الدستور.
  * محاكاة ترحيل الهياكل (Dry Run Simulation) على نسخة قاعدة بيانات محلية مصغرة ومعزولة تماماً لاختبار تماسك القيود وثبات العلاقات.

### المرحلة 5: طلب التفويض البشري (Approval Gate)
* يتوقف المحرك تماماً عن العمل، ويتم نقل حالة المهمة لـ `WaitingApproval`.
* يتم إرسال بلاغ تقني مفصل وبسيط للمستخدم عبر البريد أو لوحة تحكم التطبيق يوضح بدقة التغييرات، الأكواد، المخاطر، والفوائد.
* لا يمكن تخطي هذه البوابة أو تمرير التعديل إلا باستقبال توقيع بروتوكولي رقمي موافق عليه من المالك.

### المرحلة 6: الترحيل الفعلي والتحديث (Deployment Phase)
* بعد تلقي الموافقة، يتولى الوكيل المنفذ `Qaroni_Executor` كتابة التعديلات والمخططات على البيئة الحية بنمط التنزيل المعزول وتطبيق الأكواد في بيئة الإنتاج خطوة بخطوة بالاعتماد على بروتوكول `DeploymentProtocol`.

### المرحلة 7: الفحص البعدي والتدقيق الوقائي (Post-Verification & Rollback Gate)
* يقوم الوكيل الرقابي `Qaroni_Auditor` بإجراء فحص ومطابقة حية فورية للتأكد من نجاح تماسك الهياكل وعدم توقف الواجهات والباكيند (Smoke Test).
* في حال رصد أي خلل، تباطؤ، أو توقف في الخدمة، يتم إيقاف العملية وإلغائها فوراً ونداء بروتوكول التراجع الشامل `RollbackProtocol` لإعادة النظام للحالة الصفرية المستقرة في أجزاء من الثانية.
* يتم كتابة تقرير تدقيقي شامل وحفظه في مجلد السجلات.
