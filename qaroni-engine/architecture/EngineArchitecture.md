# Qaroni AI Engine — Architecture Specification
## المواصفة المعمارية وهندسة النظام الحاكم للمحرك الذاتي

---

## 1. بنية النظام العامة (System Architecture Layering)
تم تصميم **Qaroni AI Engine** ليعمل كـ "نظام تشغيل حاكم مصغر للـ DevOps" (Micro-Kernel Governance DevOps Engine) معزول تماماً عن كود الواجهات الأمامية والباكيند للمتجر. يتكون المحرك من خمس طبقات هيكلية أساسية:

```
┌────────────────────────────────────────────────────────┐
│               1. USER / CONTROL INTERFACE              │
│       (Approval Webhook, Triggering UI, OTP Console)   │
└───────────────────────────┬────────────────────────────┘
                            │ (Events & Action Ingress)
                            ▼
┌────────────────────────────────────────────────────────┐
│               2. QARONI GOVERNANCE CORE                │
│     (Engine Constitution, Event Bus, State Machine)     │
└───────────────────────────┬────────────────────────────┘
                            │ (Orchestrated Commands)
                            ▼
┌────────────────────────────────────────────────────────┐
│              3. SEVERE ISOLATION PIPELINE              │
│  (Knowledge Val → Supabase Branch → Docker → Production)│
└───────────────────────────┬────────────────────────────┘
                            │ (Controlled Execution Context)
                            ▼
┌────────────────────────────────────────────────────────┐
│            4. AGENT SANDBOX EXECUTION LAYER            │
│  (Reader, Analyzer, MigrationBuilder, Executor, etc.)  │
└───────────────────────────┬────────────────────────────┘
                            │ (Secure Protocols & TLS)
                            ▼
┌────────────────────────────────────────────────────────┐
│             5. PROTECTED EXTERNAL RESOURCES            │
│       (Supabase DB, GitHub Repository, Cloud Run)       │
└────────────────────────────────────────────────────────┘
```

## 2. المسار المقدس لتنفيذ العمليات (The 16-Step Execution Pipeline)

تمر كافة التغييرات والعمليات (Migration, Code Update, Configuration) بالتسلسل المقدس التالي دون أدنى استثناء:

1. **READ (القراءة):** استدعاء وقراءة التكوين، الدستور، المواصفات الحالية، وبنية قاعدة البيانات عبر الـ API.
2. **VALIDATE (المطابقة):** التحقق التلقائي والكامل لضمان عدم مخالفة الطلب لأي من بنود الدستور الحاكم.
3. **DECISION TRACE (تتبع القرار):** تسجيل أثر القرار الدستوري بالصيغة الصارمة: `Constitution (Article X) → ADR-XXX → Specification (Module Y) → Action (Migration Z)`.
4. **RISK ASSESSMENT (تقييم المخاطر):** قياس مدى خطورة التعديل (Low/Medium/High). في حال تصنيف الخطورة بـ **High**، يُجبر النظام على إيقاف التقدم التلقائي وانتظار التفويض البشري (Approval Waiting).
5. **ANALYZE (التحليل):** مقارنة البنية المطلوبة والموصوفة في الـ Specification بالمخطط الهيكلي الحالي لقاعدة البيانات.
6. **BUILD (البناء والصياغة):** صياغة ملفات الترحيل الـ SQL (Up/Down) وملفات الـ TypeScript المصاحبة.
7. **ISOLATION (العزل الصارم):** بدء تمرير التحديث في مسار العزل المتدرج (Knowledge_Validation → Supabase Branch).
8. **TEST (الاختبار):** تشغيل الفحوصات الفنية على البيئة المعزولة للتأكد من سلامة كود SQL، القيود (FK)، الفهارس، ومحاكاة عمليات الإدخال والاستعلام.
9. **SELF-VALIDATION ENGINE (محرك التحقق الذاتي):** تقييم نتائج الاختبار. إذا رصد أي فشل، يتم الانتقال تلقائياً لمرحلة الإصلاح (Feedback & Repair).
10. **FEEDBACK & REPAIR (التغذية الراجعة والإصلاح):** توليد "تقرير إصلاح" (Fix Report) يحلل سبب الفشل وإعادة المحاولة التلقائية (حتى 3 محاولات كحد أقصى).
11. **ROLLBACK (التراجع الفوري):** في حال استهلاك المحاولات الثلاث دون حل، أو في حال رصد خطر داهم، يتم تفعيل التراجع التلقائي وإلغاء المهمة فوراً.
12. **APPROVAL WAIT (انتظر التفويض البشري):** تجميد الجلسة تلقائياً وعرض الخطة التفصيلية بانتظار توقيع المالك الرقمي (OTP).
13. **EXECUTE (التنفيذ الفعلي):** تطبيق الـ Migration على قاعدة بيانات الإنتاج وتنزيل التعديلات البرمجية لـ GitHub.
14. **POST-VALIDATION (التحقق البعدي):** تشغيل الفحوصات الفنية واختبارات الأمان و الـ Smoke Run على الإنتاج لضمان الاستقرار.
15. **CLEANUP (التنظيف الشامل):** حذف الفروع التجريبية، وتصفير السجلات المؤقتة، وإغلاق بوابات العزل.
16. **AUDIT LOG (تقرير النزاهة والتدقيق):** كتابة التقرير الشامل وحفظه في مسار التدقيق الدائم لضمان الامتثال والشفافية.

## 3. آليات الأمان وقواعد قاعدة البيانات (Supabase & Database Rules)

* **الاتصال المقيد بـ REST (PostgREST Mode Only):** يحظر على كافة المكونات الاستعلام المباشر عبر بروتوكولات TCP الحرة، ويتم حصر قنوات الاتصال بالباكيند والواجهات عبر بوابة REST الآمنة لـ PostgREST.
* **إلزامية جدران الحماية (RLS Rule Enforcement):** يُمنع منعاً باتاً تمرير أو التعامل مع أي جدول لا يحتوي على سياسات أمان وصول الصفوف (Row Level Security - RLS) مفعلة ونشطة. أي جدول يخل بهذا الشرط يعتبر **BLOCKED** تلقائياً.
* **حرمة تعديلات الـ RBAC:** أي تحديث أو تعديل يمس جداول الصلاحيات والأدوار الحاكمة `rbac_*` يتطلب حصرياً تفويض الوكيل المعماري المصمم `Qaroni_Architect`.
* **انعدام SQL المباشر للإنتاج:** يمنع تشغيل استعلامات SQL حرة من قبل المستخدمين أو الوكلاء المبرمجين على بيئة الإنتاج، ويجب أن تمر كافة تعديلات المخططات حصرياً عبر ملفات الـ Migration التي تم فحصها ومحاكاتها.

---

## 4. استراتيجية العزل المتدرج (Severe Isolation Strategy)

قبل تطبيق أي تعديل على الإنتاج، يجب اجتياز البيئات المعزولة التالية بالترتيب الدقيق:

1. **Knowledge_Validation (الفحص المعرفي):** فحص منطقي وهندسي مجرد لبنية العلاقات والفهارس والتحقق من عدم تسببها في تعارضات هندسية دون لمس البيانات.
2. **Supabase Branch (فرع قواعد البيانات الافتراضي):** نسخة طبق الأصل معزولة ديناميكياً من بيئة الإنتاج الحية لإجراء فحوصات الـ SQL والـ RLS عليها حياً في بيئة سحابية معزولة.
3. **Docker Local (البيئة المحلية - اختياري):** تشغيل حاوية محلية لمحاكاة استقرار الواجهات وتجاوب الباكيند مع التعديل الهيكلي المقترح.
4. **Production (بيئة الإنتاج):** لا يتم السماح بالوصول إليها إلا بعد عبور المراحل الثلاث السابقة والتحقق من سلامتها بنسبة 100%.
