# Qaroni AI Engine — Architecture Specification
## المواصفة المعمارية وهندسة النظام الحاكم للمحرك

---

## 1. بنية النظام العامة (System Architecture Layering)
تم تصميم **Qaroni AI Engine** ليعمل كـ "نظام تشغيل حاكم مصغر" (Micro-Kernel Governance Engine) معزول تماماً عن كود الواجهات الأمامية وكود الخادم الخلفي للمشروع الحالي. يتكون المحرك من أربع طبقات هيكلية أساسية:

```
┌────────────────────────────────────────────────────────┐
│               1. USER / CONTROL INTERFACE              │
│       (Approval Webhook, Triggering UI, CLI Console)   │
└───────────────────────────┬────────────────────────────┘
                            │ (Events & Action Ingress)
                            ▼
┌────────────────────────────────────────────────────────┐
│               2. QARONI GOVERNANCE CORE                │
│     (Engine Constitution, Event Bus, State Machine)     │
└───────────────────────────┬────────────────────────────┘
                            │ (Orchestrated Commands)
                            ▼
┌────────────────────────────────────────────────────────┐
│            3. AGENT SANDBOX EXECUTION LAYER            │
│  (Qaroni Reader, Analyzer, Validator, Executor, Auditor)│
└───────────────────────────┬────────────────────────────┘
                            │ (Secure Protocols & TLS)
                            ▼
┌────────────────────────────────────────────────────────┐
│             4. PROTECTED EXTERNAL RESOURCES            │
│       (Supabase DB, GitHub Repository, Cloud Run)       │
└────────────────────────────────────────────────────────┘
```

---

## 2. المكونات المعمارية لطبقة الحوكمة (Core Architectural Layers)

### أ. طبقة الاستقبال والتحكم (Ingress & Controller Layer)
تتولى هذه الطبقة استقبال التكليفات والطلبات من واجهات النظام أو الخادم الخارجي، وتمريرها فوراً لمدير الجلسات وحالة المهام وتوليد معرف تتبع فريد للمهمة (`Task UUID`).

### ب. النواة الدستورية (The Constitutional Kernel)
تقوم بمراجعة الطلب أو الكود المقترح فحصاً أمنياً وهيكلياً قبل تفويض أي وكيل ذكاء اصطناعي (AI Agent) للتعامل معه. أي محاولة لكتابة استعلام أو إدراج تعديل يخالف القوانين الدستورية يتم رفضه وتوقيف المهمة فوراً عند هذه النواة دون الحاجة لأي تواصل خارجي.

### ج. حافلة الأحداث الداخلية (Internal Event Bus)
تدار الحركة والقرارات البرمجية داخل النواة كلياً بالاعتماد على **Event-Driven Architecture (EDA)**، مما يمنع فقدان المهام ويضمن تتبع السير الكامل للقرارات وتوثيقها بدقة متناهية غير قابلة للتلاعب أو التزييف.

### د. جدار الحماية وعزل البيانات (Double-Isolation Sandbox)
بيئة برمجية افتراضية معزولة تماماً، يتم فيها جلب بنى المخططات وتوليد الأكواد ومراجعتها ومطابقة شروط ترحيل الجداول دون مساس بالنظام الحقيقي، والتحقق الفني بـ Unit Tests مؤتمتة بنسبة 100%.

## 3. آليات الأمان والحوكمة والحد من الصلاحيات (Security Gates & Limits)
* **انعدام الوصول المباشر للعميل (Zero-Direct Access):** يمنع المحرك الوكلاء الخارجيين أو المستخدمين العاديين من الاتصال بـ PostgreSQL TCP أو إرسال استعلامات SQL حرة إلى Supabase. تمر كافة المعاملات عبر طبقة المعالجة المقيدة للمحرك.
* **الشهادات والأمينات الرقمية (Encrypted Credential Store):** تخزن مفاتيح مستودعات GitHub والـ API لـ Supabase مشفرة باستخدام خوارزمية AES-256 GCM في السيرفر ولا يمكن للوكلاء قراءتها كقيم نصية واضحة، ويتم تغذيتها حصراً عبر متغيرات البيئة المعماة أثناء التنفيذ للوظائف المصرحة.
* **الرقابة المشددة للوكيل الحارس (The Auditor Loop):** يقوم وكيل الرقابة والتدقيق المستمر بمراقبة سلوك الوكلاء الآخرين أثناء المحاكاة في الـ Sandbox، وإيقاف الجلسة ومصادرة الصلاحيات وتفعيل خطة التراجع (Rollback Protocol) عند رصد أي انحراف عن النمط المعتمد للمهمة.
* **آلية التفويض البشري بالبريد والاتصال المباشر:** العمليات الهيكلية والمحاسبية الحساسة تتطلب توقيع موافقة برمجية عبر واجهة الويب بواسطة المالك مصحوبة برمز أمان OTP أو توقيع رقمي معتمد لمنع حدوث عمليات ترحيل عشوائية ملقائية.
