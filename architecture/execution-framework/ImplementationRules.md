# 📐 قواعد وشروط التنفيذ الهيكلي (Implementation Rules - Architecture V5)

توثق هذه الوثيقة القواعد والمعايير الصارمة التي يجب اتباعها وتطبيقها حرفياً عند إجراء أي تعديل، ترحيل، أو إضافة داخل طبقة قاعدة بيانات المنصة الفيدرالية الموحدة.

---

## 🚫 1. نواهى وموانع التنفيذ الهيكلي (Strict Negative Constraints)

* **حظر الجداول العشوائية (No Unauthorized Tables):**
  يُمنع منعاً باتاً إنشاء أي جدول أو كائن مادي في قاعدة البيانات ما لم يكن معرفاً وموثقاً بالكامل وتفصيلياً داخل وثيقة المخطط العام (**Blueprint**).
* **حظر مخالفة الدستور (No Constitution Violations):**
  يُمنع تفعيل أي علاقة، أو قيد، أو سياسة أمان تخل بقواعد الدستور العليا، وبالأخص حتمية عزل بيانات الشركات المستأجرة وعدم تداخل حسابات الكوادر البشرية.
* **حظر حذف الكيانات التشغيلية والمالية (No Hard Deletion for Financial/Operational Entities):**
  يُمنع منعاً باتاً حذف (Hard Delete) أي صف أو سجل مالي أو حركي يتبع لجداول المبيعات، الفواتير، الحسابات، الدفتر المالي العام، الصناديق، أو المخازن. يتم الاعتماد حصرياً على الحذف المنطقي (Soft Delete) أو إقفال الحالات للحفاظ على التاريخية والأثر المالي والضريبي للمنصة.
* **حظر إنشاء المشغلات العشوائية (No Unregulated Triggers/Functions):**
  يُمنع بناء أي مشغل تلقائي بالخلفية (`TRIGGER`) أو دالة مخزنة (`FUNCTION`) خارج الأوصاف الفنية وشبكة العلاقات المدونة في عقود التنفيذ المعتمدة (**Execution Contracts**).
* **حظر العبث بجدول التدقيق التاريخي (No Auditing Tampering):**
  يُمنع بتاتاً السماح لأي مستخدم أو موظف أو مدير نظام بتعطيل، أو تعديل، أو حذف صفوف جدول الرقابة والتدقيق التاريخي الموحد `audit_log` تحت طائلة قفل الحساب والمحاسبة المعمارية.

## ⚙️ 2. القواعد والمحددات التنفيذية الإلزامية (Mandatory Design Practices)

### أ. الهوية الرقمية وعزل البيانات (Multi-tenancy & Identity)
* يجب أن يحتوي كل جدول تشغيلي، مرجعي، أو مالي على حقل المعرف الفريد للشركة المستأجرة `organization_id` لضمان عزل البيانات الحاسم.
* تلتزم كافة الجداول بتهيئة المعرفات الفريدة باستخدام نوع البيانات `UUID` مولداً بطريقة معماة وغير قابلة للتخمين (مثل استخدام `gen_random_uuid()` أو `uuid_generate_v4()`).

### ب. تكامل القيود والتحققات الحركية (Integrity Checks)
* يجب ربط قيود التحقق (`CHECK Constraints`) على كافة الحقول المالية والكمية لضمان عدم قبول قيم سالبة (مثال: الأسعار، الكميات، الضرائب، أرصدة العهد النقدية).
* فرض القيود المدمجة والفريدة لرموز السلع والمنتجات (SKU) والباركودات البديلة، مع تصفيتها وعزلها الصارم على مستوى المنظمة لعدم تداخل أكواد المنتجات عبر الشركات المستأجرة.

### ج. حماية وتجميد القيود المالية (Financial Immutability)
* تُصاغ الدوال والمشغلات بصرامة تمنع تعديل أو حذف القيود المحاسبية للدفتر العام وفواتير المبيعات الصادرة المسددة بعد ترحيلها التاريخي، واعتماد سياسة الإلغاء بالسندات العكسية والتسويات الموازية فقط.

### د. حوكمة آلات الحالات (State Machines Enforcement)
* تخضع معاملات الفواتير، النقل البيني، والديون لرقابة آلات الحالات المبرمجة؛ ويُمنع تخطي أي خطوة مراجعة أو ترحيل أو تسيير خارج الأطر الانتقالية المحددة بجدول مسارات العمل المعزول.
