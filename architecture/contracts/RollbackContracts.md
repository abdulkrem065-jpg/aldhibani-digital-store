# 🔄 عقود بروتوكول التراجع السليم (Rollback Contracts - Architecture V5)

يحدد هذا المستند الشروط والمعايير الهندسية الصارمة للتراجع الفني الآمن عن الترحيلات الهيكلية لقاعدة البيانات (Database Rollback)، لضمان عدم حدوث تشويه أو فقدان عشوائي للبيانات وحفظ سلامة النظام التاريخية والأمنية.

---

## 1. محددات الترخيص وحظر التراجع (Rollback Permission Gate)

قبل البدء في تنفيذ أي عملية تراجع فني، يجب إخضاع الحركة للفحص الشديد لتحديد ما إذا كان التراجع مسموحاً به هندسياً أم يمثل خطراً جسيماً على سلامة البيانات وصحتها التاريخية.

### 🟢 متى يكون التراجع مسموحاً به؟ (Permitted Rollback)
1. **أثناء مرحلة التطوير والاختبار (Staging & Testing Phase):** يسمح بالتراجع الشامل لتصحيح الأخطاء الهيكلية والفهارس وتحسين الأداء.
2. **فشل الفحوصات الفنية لبروتوكول التحقق (Verification Failure):** إذا فشل ملف ترحيل فني في اجتياز أي من شروط العبور الهيكلية أو الوظيفية أو الأمنية في بيئة الاستعداد، فيجب التراجع عنه فورا لتصحيح الخلل.
3. **التحديثات الهيكلية المنعزلة (Isolated Schema Updates):** في حال التحديثات الهيكلية التي لا تمس بيانات العملاء أو السجلات المالية التاريخية الحية بنطاق مباشر.

### 🔴 متى يمنع التراجع نهائياً؟ (Strictly Forbidden Rollback)
1. **وجود بيانات حية في الإنتاج (Live Production Data Exist):** يمنع تماماً التراجع التلقائي بالحذف (Drop) عن الجداول المالية أو جداول العملاء أو المبيعات بعد دخول بيئة الإنتاج الفعلي وبلوغ العمليات حيز التنفيذ التجاري.
2. **القيود المحاسبية المرحلة والمعتمدة (Posted Ledger Entries):** يحظر إسقاط أو إفراغ جداول الدفتر العام والقيود اليومية `ledger_entries` لمنع محاولات طمس الحقائق وتجاوز التدقيق الضريبي، ويجب معالجة الأخطاء المحاسبية بقيود عكسية متوازنة فقط (Reversing Entries) دون المساس بهياكل الجداول.
3. **سجلات الرقابة الفيدرالية الموحدة (Audit Logs):** يمنع نهائياً التراجع التدميري أو الحذف لجدول الرقابة `audit_log` تحت أي مسمى؛ فالجدول محكوم بمبدأ الـ (Append-Only) ولا يسمح بأي تراجع هيكلي يؤدي لطمس الأثر الرقابي للفاعلين البشريين بالنظام.

## 2. ترتيب التراجع السليم وعكس المسار الحرج (Rollback Order)

عند الموافقة الفنية المعتمدة على التراجع عن مرحلة هيكلية معينة؛ يجب إخضاع الكائنات للترتيب العكسي التام والدقيق للإنشاء (Reverse Topological Order)، لضمان عدم بقاء هياكل معطوبة أو التسبب في فشل القيود المتبادلة.

```text
[الإنشاء الهيكلي]:  01.المنظمات ──► 02.الهوية ──► 05.المخازن ──► 06.المبيعات ──► 07.المحاسبة ──► 10.الرقابة
[التراجع الفني]: 10.الرقابة  ──► 07.المحاسبة ──► 06.المبيعات ──► 05.المخازن ──► 02.الهوية ──► 01.المنظمات
```

| الترتيب (Rollback Order) | المجال المستهدف (Target Domain) | الكائنات المتأثرة بالاستبعاد (Dropped Objects) |
| :---: | :--- | :--- |
| **1** | **الذكاء الاصطناعي (AI)** | حجب سياسات RLS، إسقاط الجداول `ai_memory`, `ai_knowledge_base`, `ai_messages`, `ai_conversations` |
| **2** | **التنبيهات (Notification)** | حجب سياسات RLS، إسقاط الجداول `user_notification_prefs`, `notification_templates`, `notifications` |
| **3** | **الرقابة (Audit)** | حجب سياسات RLS، إسقاط الجداول والمشغلات المرتبطة بـ `audit_log` (يمنع بالإنتاج) |
| **4** | **المحاسبة (Accounting)** | حجب سياسات RLS، إسقاط الجداول `debts`, `money_boxes`, `ledger_entries`, `accounts` (يمنع بالإنتاج) |
| **5** | **المبيعات (Sales)** | حجب سياسات RLS، إسقاط الجداول `pending_carts`, `order_items`, `orders` (يمنع بالإنتاج) |
| **6** | **مسارات العمل (Workflow)** | حجب سياسات RLS، إسقاط الجداول `workflow_transitions`, `workflow_states` |
| **7** | **المخزون (Inventory)** | حجب سياسات RLS، إسقاط الجداول `stock_adjustments`, `stock_transfers`, `branch_inventory` |
| **8** | **المنتجات (Product)** | حجب سياسات RLS، إسقاط الجداول `product_barcodes`, `products`, `categories` |
| **9** | **العملاء (Customer)** | حجب سياسات RLS، إسقاط الجداول `customer_loyalty`, `customer_profiles`, `customers` |
| **10** | **الصلاحيات (RBAC)** | حجب سياسات RLS، إسقاط الجداول `rbac_staff_user_roles`, `rbac_role_permissions`, `rbac_permissions`, `rbac_roles` |
| **11** | **الهوية (Identity)** | حجب سياسات RLS، إسقاط الجداول `organization_members`, `staff_users` |
| **12** | **المنظمات (Organization)** | حجب سياسات RLS، إسقاط الجداول `store_config`, `branches`, `organizations` |

## 3. عقيدة الورقة البيضاء وحصانة الكائنات (Clean Slate vs. Untouchable Objects)

عند تفعيل بروتوكول التراجع الشامل لإعادة تهيئة النظام لبيئة التطوير (Database Clean Slate)، يجب توضيح وتصنيف الكائنات بحزم:

### 🧹 الكائنات المسموح بإسقاطها وإعادة تهيئتها (Clean Slate Objects)
* جداول المبيعات المؤقتة وسلال المشتريات المعلقة للعملاء `pending_carts`.
* جداول التنبيهات والإشعارات الفردية `notifications` وتفضيلات الإرسال.
* جداول محادثات ورسائل الذكاء الاصطناعي وجلسات الدعم الفني المخصصة.
* جداول تنظيم مسارات العمل الانتقالية وحالات الفواتير والعمليات غير المرحلة.
* فهارس البحث والتطابق الفردية والمدمجة، والدوال البرمجية المساعدة لتحديث أوقات التعديل والتوليد التلقائي للنصوص.

### 🛡️ الكائنات المحصنة والمنيعة ضد التراجع التلقائي (Untouchable Objects)
* **سجل الرقابة والتدقيق التاريخي الموحد (`audit_log`):** يمنع تدميره أو حذف محتواه، لحمايته كأصل فيدرالي معتمد لإثبات الشفافية وتدقيق تاريخ العمليات بالنظام.
* **القيود والدفاتر اليومية المعتمدة محاسبياً والديون (`ledger_entries`, `debts`):** لا يسمح بالتراجع الحذفي للقيود التي دخلت حيز الحسابات الرسمية، حماية للنزاهة الضريبية ومنعاً للثغرات الحسابية والتداخل التمويلي.
* **ملفات وبيانات الهوية الحقيقية ومستأجري الكيانات العليا (`organizations`):** حظر الحذف الشامل لملفات المستأجرين النشطين والمسجلين قانونياً لمنع تعطل المنصة التجاري.
