# 📊 مخطط مجال التدقيق وسجلات الرقابة الفيدرالية (Audit Database Blueprint)

## 1. Purpose (الهدف الفني)
إنشاء وبناء السجل الفيدرالي التاريخي الرقابي الموحد والأمين للعمليات في قاعدة البيانات، وحظر تام ومطلق للتعديل أو الحذف بجدول التدقيق، وإرساء محركات الـ Triggers التلقائية لالتقاط لقطات البيانات قبل التعديل وبعده لضمان المطابقة والأمن والامتثال القانوني.

---

## 2. Tables (الجداول التابعة للمجال)
* `audit_log` (سجل الرقابة الفيدرالي والتدقيق الموحد للنظام)

---

## 3. Columns (الأعمدة وهياكلها)

### جدول `audit_log`
| Column | Type | Nullable | Default | Notes |
| :--- | :--- | :--- | :--- | :--- |
| `id` | UUID | No | gen_random_uuid() | Primary Key |
| `organization_id` | UUID | No | None | Foreign Key للمنظمة المستأجرة المتأثرة |
| `branch_id` | UUID | Yes | NULL | Foreign Key للفرع المنفذ تحته الحركة |
| `staff_id` | UUID | Yes | NULL | Foreign Key للموظف البشري الفاعل للحركة (Actor) |
| `table_name` | VARCHAR(100) | No | None | اسم الجدول التشغيلي الذي تأثر بالعملية (مثل: orders, products) |
| `action_type` | VARCHAR(50) | No | None | نوع العملية في قاعدة البيانات (INSERT, UPDATE, DELETE) |
| `record_id` | UUID | No | None | معرف السجل الفردي والفيزيائي المتأثر بالحركة |
| `old_data_meta` | JSONB | Yes | NULL | لقطة من الحالة القديمة للسطر قبل التغيير |
| `new_data_meta` | JSONB | Yes | NULL | لقطة من الحالة الجديدة للسطر بعد التغيير |
| `ip_address` | VARCHAR(45) | Yes | NULL | عنوان بروتوكول الإنترنت للمنفذ إن وجد |
| `created_at` | TIMESTAMPTZ | No | CURRENT_TIMESTAMP | تاريخ ووقت التقاط التغيير الفعلي بدقة ميكروثانية |

---

## 4. Keys (المفاتيح والمؤشرات)
* **Primary Keys:**
  * `audit_log.id`
* **Foreign Keys:**
  * `audit_log.organization_id` يشير إلى `organizations.id` (مع CASCADE).
  * `audit_log.branch_id` يشير إلى `branches.id` (مع SET NULL عند الحذف).
  * `audit_log.staff_id` يشير إلى `staff_users.id` (مع SET NULL عند الحذف لتبقى السجلات التاريخية سليمة).

---

## 5. Relationships (العلاقات ورسمها النصي)
* **سجل الرقابة الفيدرالي:** يرتبط بالمنظمة، الفرع، والموظف البشري الفاعل للعملية لتوثيق المسؤولية والمساءلة الإدارية.

```text
[ organizations ] ◄─── 1:N ───┐
[ branches ] ◄──────── 1:N ───┼─── [ audit_log ]
[ staff_users ] ◄───── 1:N ───┘
```

---

## 6. Constraints (القيود الهندسية)
* **NOT NULL:** مطبق على معرف السجل، اسم الجدول، نوع العملية، المنظمة الأم، وتواريخ الحفظ والتقاط الحركة.
* **CHECK:**
  * نوع العمليات يقتصر على الحركات الأساسية لقاعدة البيانات (`action_type` ينتمي إلى: INSERT, UPDATE, DELETE).

---

## 7. Index Strategy (استراتيجية الفهارس)
* **BTree Index:**
  * فهرس مدمج على `audit_log(organization_id, table_name, created_at)` لتسريع واجهات وتقارير الرقابة والامتثال للمسؤولين.
  * فهرس تجميعي على `audit_log.record_id` لتسريع استدعاء وعرض تاريخ سطر أو فاتورة معينة بالكامل.
  * فهرس تجميعي على `audit_log.staff_id` لمتابعة حركات موظف معين.

---

## 8. Generated Columns (الأعمدة المولدة تلقائياً)
* لا توجد أعمدة مولدة حالياً في جدول سجلات الرقابة.

---

## 9. Computed Fields (الحقول المحسوبة برمجياً)
* لا توجد حقول محسوبة برمجياً بجدول الرقابة؛ حيث يتم الاحتفاظ بالبيانات خاماً ومطابقة للحركات الفيزيائية لمنع أي خلل بالتأويل والتوثيق.

---

## 10. Triggers Required (المؤشرات والمشغلات المطلوبة)
* `block_audit_log_mutation`:
  * **الهدف:** القفل التام والحظر المطلق لأي تعليمة لتعديل أو حذف أو تصفير السجلات التاريخية في جدول `audit_log` من قبل أي مستخدم أو مشرف مهما علت رتبته (Append-Only الصارم).
  * **متى يعمل:** قبل التعديل أو الحذف (`BEFORE UPDATE OR DELETE`).

---

## 11. Functions Required (الدوال المطلوبة)
* `audit_generic_row_mutation`:
  * **الهدف:** دالة مركزية لالتقاط الفوارق والبيانات وتدقيق حركة الجداول الحساسة وتجهيز لقطات الـ JSONB القديمة والجديدة لترحيلها بجدول الرقابة تلقائياً.
  * **Input:** `OLD_RECORD` (RECORD), `NEW_RECORD` (RECORD).
  * **Output:** TRIGGER (لحفظ السجل).

---

## 12. RLS Strategy (استراتيجية عزل الضمان وبوابة الأمان)
* **العزل الفيدرالي الحاسم (Tenant Dimension):**
  * يخضع جدول التدقيق لعزل المنظمة والأمان الفيدرالي الصارم.
  * تُقفل الصلاحيات على جداول الرقابة تماماً بسياسات تمنع عمليات (`UPDATE`, `DELETE`) لكافة مستخدمي قاعدة البيانات، وتقتصر الصلاحيات على عمليات الإضافة التلقائية من محركات الجداول وقراءتها فقط للمسؤولين والمدراء المعتمدين بموجب سجل الـ RBAC.

---

## 13. Audit Requirements (متطلبات الرقابة والتدقيق التاريخي)
* جدول التدقيق يمثل المرجع والعماد لرقابة النظام بأكمله؛ لذلك هو خارج نظام الفرز الذاتي لمنع الدوران اللا نهائي للـ Triggers، ويتم حراسته أمنياً وهندسياً عبر قيود قاعدة البيانات الصارمة.

---

## 14. AI Integration (التكامل مع الذكاء الاصطناعي وحماية البيانات)
* **ما يرسل:** يرسل للذكاء الاصطناعي سجل حركات الموظفين ومعدل التغييرات بالدورة المالية والمخازن لاكتشاف الأخطاء وتسهيل المتابعة الرقابية والمساءلة.
* **ما يمنع إرساله:** يمنع كلياً تزويد الذكاء الاصطناعي بكلمات المرور القديمة أو تفاصيل البطاقات الائتمانية حتى لو تم رصد تعديلها في لقطات التدقيق.

---

## 15. Events (الأحداث المصدرة والمستقبلة)
* **الأحداث المصدرة:**
  * `audit.entry.logged` (عند تدوين حركة جديدة لتحديث واجهات الرقابة المباشرة للمشرفين).
  * `audit.mutation.suspicious` (عند رصد تكرار حركات الحذف أو التعديل على الجداول المالية لطلب التحقق البشري الفوري).

---

## 16. Performance Considerations (اعتبارات الأداء والتحسين الفيدرالي)
* جدول الرقابة يعتبر الجدول الأكبر حجماً والأعلى كتابة (Write-Heavy) في قاعدة البيانات مع مرور الوقت. ينصح بشدة بالتقسيم والفرز التاريخي الجدولي (`Table Partitioning`) شهرياً أو سنوياً بموجب حقل `created_at` لضمان عدم بطء الاستعلامات واستدامة النظام.

---

## 17. Future Expansion (خطط التوسيع المستقبلية والترقية الآمنة)
* التخطيط لترحيل وتصدير الأرشيف التدريبي والرقابي القديم تلقائياً إلى وحدات تخزين سحابية مغلقة وباردة (Cold Storage) بعد فترات محاسبية معينة، لتقليص مساحات الأقراص السحابية والحفاظ على خفة وسرعة قاعدة البيانات الأساسية.
