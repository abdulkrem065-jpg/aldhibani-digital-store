# ⚙️ مخطط مجال مسارات العمل وآلات الحالات (Workflow Database Blueprint)

## 1. Purpose (الهدف الفني)
تنظيم وضبط وإدارة الانتقالات والحالات الهيكلية المنطقية للكيانات المعقدة، ومنع الحالات العشوائية أو التناقض الانتقالي للجداول الحساسة (مثل الطلبات والديون)، وفرض آلات الحالات الصارمة على الجداول لضمان جودة المسارات التشغيلية والمالية.

---

## 2. Tables (الجداول التابعة للمجال)
* `workflow_states` (الحالات والمراحل المعرفة بالمسار)
* `workflow_transitions` (الانتقالات والقواعد المنطقية الحاكمة)

---

## 3. Columns (الأعمدة وهياكلها)

### جدول `workflow_states`
| Column | Type | Nullable | Default | Notes |
| :--- | :--- | :--- | :--- | :--- |
| `id` | UUID | No | gen_random_uuid() | Primary Key |
| `organization_id` | UUID | No | None | Foreign Key للمنظمة المستأجرة |
| `entity_type` | VARCHAR(100) | No | None | نوع الكيان المطبق عليه المسار (مثل: order, debt) |
| `state_key` | VARCHAR(100) | No | None | الرمز الفني للحالة (مثل: draft, pending_review, completed, cancelled) |
| `state_label` | VARCHAR(150) | No | None | الاسم والتوصيف المقروء للحالة (مثل: مسودة، بانتظار المراجعة) |
| `is_terminal` | BOOLEAN | No | FALSE | مؤشر ما إذا كانت الحالة نهائية ومغلقة تمنع أي تعديل لاحق للسطر |

### جدول `workflow_transitions`
| Column | Type | Nullable | Default | Notes |
| :--- | :--- | :--- | :--- | :--- |
| `id` | UUID | No | gen_random_uuid() | Primary Key |
| `organization_id` | UUID | No | None | Foreign Key للمنظمة المستأجرة |
| `entity_type` | VARCHAR(100) | No | None | نوع الكيان المطبق عليه الانتقال الحاكم |
| `from_state_key` | VARCHAR(100) | No | None | الرمز الفني للحالة الحالية والمنطلق منها |
| `to_state_key` | VARCHAR(100) | No | None | الرمز الفني للحالة المستهدفة والمطلوب الانتقال إليها |
| `required_permission` | VARCHAR(150) | Yes | NULL | الصلاحية الذرية المطلوبة في الـ RBAC للمصادقة على الانتقال |

---

## 4. Keys (المفاتيح والمؤشرات)
* **Primary Keys:**
  * `workflow_states.id`
  * `workflow_transitions.id`
* **Foreign Keys:**
  * `workflow_states.organization_id` يشير إلى `organizations.id` (مع CASCADE).
  * `workflow_transitions.organization_id` يشير إلى `organizations.id` (مع CASCADE).
* **Candidate/Unique Keys:**
  * `workflow_states` (مزيج `organization_id` + `entity_type` + `state_key` فريد لمنع تكرار تعريف الحالات لنفس الكيان بالمستأجر).
  * `workflow_transitions` (مزيج `organization_id` + `entity_type` + `from_state_key` + `to_state_key` فريد لضمان تعريف قاعدة انتقال واحدة فقط للمسار).

---

## 5. Relationships (العلاقات ورسمها النصي)
* **هيكل آلة الحالات (State Machine):** يربط الانتقال بالحالة الحالية والحالة المستهدفة المعتمدة لتسهيل الفحص والتحقق من سلامة الطلب.

```text
[ workflow_states (البداية) ] ◄── from_state_key ──┐
                                                   ├───► [ workflow_transitions ]
[ workflow_states (الهدف) ] ◄─── to_state_key ───┘
```

---

## 6. Constraints (القيود الهندسية)
* **NOT NULL:** مطبق على معرفات الجداول، الكيانات، الرموز، والحالات والانتقالات.
* **CHECK:**
  * خلو الرموز الفنية `state_key` و `from_state_key` من الفراغات لضمان سلامة المطابقة.
  * عدم تماثل بداية ونهاية الانتقال (`from_state_key` لا يساوي `to_state_key` لمنع الدوران اللا نهائي).

---

## 7. Index Strategy (استراتيجية الفهارس)
* **BTree Index:**
  * فهرس مدمج وفريد على `workflow_states(organization_id, entity_type, state_key)` لتسريع التحقق من صحة الحالة للكيان.
  * فهرس مدمج وفريد على `workflow_transitions(organization_id, entity_type, from_state_key, to_state_key)` لتسريع مطابقة القواعد المسموحة للانتقال.

---

## 8. Generated Columns (الأعمدة المولدة تلقائياً)
* لا توجد أعمدة مولدة حالياً في جداول مسارات العمل.

---

## 9. Computed Fields (الحقول المحسوبة برمجياً)
* لا توجد حقول محسوبة برمجياً بجدول مسارات العمل؛ حيث يتم الاحتفاظ بقيم ومفاتيح الحالات كمعايير حتمية وثابتة في قاعدة البيانات.

---

## 10. Triggers Required (المؤشرات والمشغلات المطلوبة)
* `assert_valid_state_transition`:
  * **الهدف:** التحقق والحظر الفوري من قيام الباكيند أو الموظف بتعديل حقل الحالة لطلب أو دين ما لم يكن الانتقال مسجلاً ومسموحاً به في جدول القواعد `workflow_transitions`.
  * **متى يعمل:** قبل تحديث الكيانات المرتبطة بالمسارات (`BEFORE UPDATE` على الكيانات الخارجية).

---

## 11. Functions Required (الدوال المطلوبة)
* `validate_transition_logic`:
  * **الهدف:** دالة فحص وتأكيد منطقية ترجع نعم أو لا بعد مطابقة صلاحيات المستخدم الفاعل مع الصلاحية المطلوبة للانتقال `required_permission`.
  * **Input:** `org_id` (UUID), `entity` (VARCHAR), `from_state` (VARCHAR), `to_state` (VARCHAR), `staff_user_id` (UUID).
  * **Output:** BOOLEAN.

---

## 12. RLS Strategy (استراتيجية عزل الضمان وبوابة الأمان)
* **العزل الفيدرالي للمسارات (Tenant Dimension):**
  * تخضع جميع جداول مسارات العمل لعزل المنظمة والأمان الفيدرالي.
  * تُقفل جداول مسارات العمل وصياغة القواعد بقوانين تمنع التعديل والإضافة لغير المشرفين ومدراء النظام بالشركة، وتقتصر عامة الموظفين على القراءة المرجعية فقط لضمان سلامة واستقرار آلات الحالات.

---

## 13. Audit Requirements (متطلبات الرقابة والتدقيق التاريخي)
* يُخضع جدول الحالات والانتقالات لتدقيق ومتابعة حركات المشرفين لضمان عدم تخريب أو إلغاء قيود آلات الحالات والالتفاف على دورات حياة المستندات والطلبات المالية بالشركة.

---

## 14. AI Integration (التكامل مع الذكاء الاصطناعي وحماية البيانات)
* **ما يرسل:** يرسل للذكاء الاصطناعي خريطة وحالات مسارات العمل المتاحة لمساعدته في توجيه الموظفين وإرشادهم للمرحلة والخطوة التالية المطلوبة لمعالجة الطلبات.
* **ما يمنع إرساله:** يمنع تزويد الذكاء الاصطناعي بأي قدرة لتجاوز آلة الحالات أو تعديل قواعد الانتقال المعتمدة يدوياً.

---

## 15. Events (الأحداث المصدرة والمستقبلة)
* **الأحداث المصدرة:**
  * `workflow.state.transitioned` (عند اكتمال انتقال سليم للكيان لإطلاق وحث الأنظمة الفرعية لتفعيل وظائفها).
  * `workflow.transition.blocked` (عند رصد محاولة تلاعب وتجاوز لآلة الحالات لتنبيه أنظمة الرقابة والأمن).

---

## 16. Performance Considerations (اعتبارات الأداء والتحسين الفيدرالي)
* جداول آلات الحالات وقواعدها تعتبر صغيرة الحجم وخفيفة للغاية وعالية القراءة؛ لذلك يوصى بقوة بالاحتفاظ بفهارسها نشطة لتقليل زمن استعلام الفحص قبل كل عملية تعديل للكيانات.

---

## 17. Future Expansion (خطط التوسيع المستقبلية والترقية الآمنة)
* إمكانية ربط مسارات العمل بمحركات التوقيع المزدوج أو الموافقات المتعددة (Multi-Signature Workflow Approvals) مستقبلاً، عبر التخطيط لإضافة جدول روابط الموافقات دون تدمير آلات الانتقال الثابتة والقائمة بنطاق عزل الـ RLS.
