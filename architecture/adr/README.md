# 🧭 سجل القرارات المعمارية (Architectural Decision Records - ADR)

## 🎯 هدف المجلد
هو المستودع الفني المرن الذي يوثق تاريخ وحيثيات وسياق القرارات الهندسية والتقنية الهامة التي تم اتخاذها أثناء تطوير النظام. يضمن هذا المجلد بقاء أسباب القرارات الكبرى واضحة للمطورين الحاليين والمستقبليين وحمايتها من النسيان أو التحريف.

## 🛠️ وظيفة الملفات الموجودة بداخله
* **ملفات الـ ADR المعنونة** (مثل `0001-v5-execution-roadmap.md`): يسجل كل ملف قراراً هندسياً محدداً عبر الهيكل القياسي للأدر: (الاسم، الحالة، السياق، القرار المعين، العواقب والمخاطر المترتبة).
* **`README.md`**: يوفر الفهرس العام لكافة القرارات النشطة وحالتها (نشط/ملغى/مستبدل).

## ✅ ما يجوز وضعه بداخله
* وثائق سجل القرارات المعمارية الجديدة بصيغة ممارسات الـ ADR القياسية.
* المخططات الهندسية التوضيحية أو الرسوم البيانية المساعدة في تبرير الخيارات المعمارية المقترحة.
* سجلات المقارنات التقنية بين أدوات أو هياكل برمجية معينة.

## ❌ ما يُمنع وضعه بداخله
* يُمنع منعًا باتًا وضع ملفات الكود المصدري أو سكربتات SQL الحية للتنفيذ أو التعديل.
* يُمنع وضع وثائق القواعد المعمارية الثابتة الكبرى (مكانها مجلد `constitution`).

## 🔄 ترتيب استخدامه أثناء دورة التطوير
يأتي سجل الـ ADR كـ **الخطوة الثانية (`Step 2`)** مباشرة بعد مراجعة الدستور:
1. عند ظهور تحدٍ هندسي أو رغبة في بناء ميزة معمارية، يفتح المهندس ملف ADR جديد للمناقشة وتبرير الخيارات.
2. يتم موازنة الخيارات المتاحة ومطابقتها مع الدستور الأعلى للتأكد من عدم وجود تعارض.
3. بمجرد اعتماد القرار وتوقيعه، يتم ترقيته إلى حالة `Accepted` أو `Active` والبدء بصياغة سكربت الامتثال والتنفيذ بناءً عليه.
