## المادة الأولى: المعايير الهندسية وجودة الأكواد (Article I: Engineering Standards & Quality)

1.1. **الالتزام بالهيكل أحادي الاتجاه (Unidirectional Layered Architecture):**  
تتبع جميع البرمجيات والشيفرات داخل نظام قاروني هيكلاً طباقياً واضحاً وصارماً. يُمنع منعاً باتاً استدعاء الطبقات العليا من قبل الطبقات السفلى لضمان استقرار النظام ومنع التشابك والتعقيد الهندسي.

1.2. **استخدام لغة TypeScript المطورة (Strict Type Safety Rules):**  
كل الكود البرمجي يجب أن يُكتب باستخدام TypeScript الصارمة بدون استخدام أنواع مرنة أو مجهولة (`any`) إلا في الحالات المحددة استثنائياً بترخيص أمني. يجب تمرير فحص Linter و Type-checker بنسبة نجاح 100% قبل البناء.

1.3. **الاعتماد على المكتبات والحلول الأصلية (Standard Library Reliance):**  
يُحظر اختراع دوال أو مكتبات فرعية جديدة للقيام بمهام توفرها لغة البرمجة أو إطار العمل بالفعل. يُمنع تكرار الوظائف ومحركات الحوسبة وتشتيت المنطق البرمجي.

## المادة الثانية: قواعد اختبار وسلامة الأنظمة (Article II: Testing & Code Verification)

2.1. **إلزامية تشغيل الفحوصات الجافة والآلية (Dry Run Verification):**  
قبل ترحيل أو دمج أي شيفرة برمجية جديدة لمستودع الإنتاج، يجب إخضاع الكود الجديد لفحص جاف واختباري داخل بيئة معزولة (Dry Run). يتم فحص كفاءة الكود والتأكد من خلوه من الأخطاء التكرارية.

2.2. **حلقة التعافي الذاتي من أخطاء البناء (Autonomous Code Self-Healing Loop):**  
في حال واجه النظام أخطاء أثناء البناء والترجمة البرمجية، يتم تشغيل حلقة التصحيح التلقائي للذكاء الاصطناعي (حتى 3 محاولات كحد أقصى) لتعديل صياغة الأخطاء آلياً وإعادة التحقق، وفي حال فشل المحاولات يتوقف المسار كلياً ويتم إحالة الأمر للمطور البشري.

2.3. **توليد بصمة المكون البرمجي (Component Integrity Hash):**  
كل تعديل برمجي معتمد يحصل على بصمة تجزئة أمنية فريدة (SHA-256 Hash) لضمان نزاهة الملفات ومنع التلاعب الخارجي بالملفات أثناء التشغيل.

## المادة الثالثة: حوكمة التوثيق والشيفرة النظيفة (Article III: Documentation & Clean Code)

3.1. **إلزامية التوثيق الموازي (Parallel Documentation Rule):**  
لا تُقبل أي ميزة برمجية جديدة أو ترقية للهيكل البرمجي دون أن يتم توثيقها بشكل فوري في ملفات التوضيح والكتالوجات الخاصة بالنظام، مع ربطها بسلسلة تتبع القرار المعماري (Decision Trace).

3.2. **المحافظة على نظافة الملفات المصدرية (Anti-Clutter Clause):**  
يجب إزالة كافة الأكواد المعطلة وغير المستخدمة أو التعليقات المؤقتة القديمة ومذكرات العمل لضمان بقاء الملفات البرمجية نظيفة وقابلة للقراءة بوضوح تام من المطور البشري والوكلاء على حد سواء.
